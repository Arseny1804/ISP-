# Подробный отчёт об эксперименте

## 1. Выбор проекта

Изначально рассматривались **bftpd** и **ProFTPD** (FTP-серверы).
Оба варианта были отклонены после проверки: оба являются
официальными субъектами ProfuzzBench (наравне с LightFTP и
Pure-FTPd), а ProFTPD дополнительно фигурирует в оригинальных
статьях AFLNet и StateAFL (последняя нашла в нём heap
buffer over-read через собственный аллокатор памяти пулов), а также
в отдельном разборе GitHub Security Lab по фаззингу FTP-серверов.
Оба факта напрямую нарушают условия задания "не входит в
ProfuzzBench" и "нет готовых материалов по фаззингу".

Взамен выбран **ngIRCd** — IRC-сервер на C. Обоснование см. в
README.md. Дополнительно: ngIRCd — однопроцессный сервер
(select/epoll event loop без форка на соединение), что упрощает
работу с классическим AFL forkserver по сравнению с
мультипроцессными демонами вроде ProFTPD.

## 2. Среда

- Ubuntu 26.04 LTS ("Resolute Raccoon"), WSL2 под Windows 10 Pro
- AFL++ (afl-cc 4.33c) установлен системно, доступен как afl-clang-fast
- clang 21.1.8, gcc/gcov 15.2.0
- lcov 2.4-3

## 3. Сборка ngIRCd обычным способом

```bash
git clone https://github.com/ngircd/ngircd.git
cd ngircd
sudo apt install -y build-essential autoconf automake pkg-config libssl-dev zlib1g-dev
./autogen.sh
./configure
make -j$(nproc)
```

Собралось без ошибок (только предупреждения -Wdiscarded-qualifiers).

## 4. Ручная проверка протокола (основа для seed)

Минимальный конфиг для тестового стенда (fuzz/config/ngircd_6667.conf):

```
[Global]
Name = fuzz0.test.server
Ports = 6667
Listen = 127.0.0.1

[Limits]
MaxConnectionsIP = 0
PingTimeout = 300
PongTimeout = 300

[Options]
DNS = no
Ident = no
PAM = no
```

DNS/Ident отключены, чтобы сервер не делал внешних lookup-запросов
на каждое соединение (иначе — недетерминизм и задержки при фаззинге).
MaxConnectionsIP=0 понадобился позже, при массовом replay corpus'а
через coverage-сборку (см. раздел 8) — иначе сервер отбрасывает
подключения с одного IP при лимите по умолчанию (5).

Ручная сессия через nc подтвердила полный stateful-хендшейк:
PASS/NICK/USER -> numeric-ответы 001-255 -> JOIN -> 353/366 ->
PRIVMSG -> PART -> QUIT -> ERROR/закрытие соединения.
Дамп этой сессии сохранён как seeds/seed1_full_session.raw
(с CRLF-терминаторами по RFC 1459).

## 5. Установка AFLNet и первая проблема сборки

```bash
git clone https://github.com/aflnet/aflnet.git
cd aflnet
make clean && make
```

**Проблема:** классический afl-gcc режим требует компиляции
32-битного кода для проверки; на Ubuntu 26.04 без gcc-multilib
это падает с ошибкой `test_x86`.

**Решение:** LLVM-режим (afl-clang-fast) не требует 32-бит,
поэтому проверку можно пропустить: `AFL_NO_X86=1 make`.

**Проблема 2:** сборка падает в afl-fuzz.c:
`'FALSE' undeclared` / `'TRUE' undeclared` в функции
update_state_aware_variables (вызов agnode()). Причина —
новая версия libgraphviz-dev (14.1.2 в Ubuntu 26.04) больше
не определяет макросы TRUE/FALSE в cgraph.h, а код AFLNet
2020 года их использует напрямую.

**Решение** (patches/0001): добавлены собственные `#ifndef`-защищённые
определения TRUE/FALSE в начало afl-fuzz.c.

После этого `AFL_NO_X86=1 make` собирает classic afl-fuzz,
aflnet-replay, afl-showmap, afl-tmin и т.д. без ошибок.

## 6. Добавление поддержки протокола IRC в AFLNet

AFLNet поддерживает разбор состояний сервера (флаг -P) только
для 17 встроенных протоколов (см. README.md), IRC среди них нет.
Это задокументированная точка расширения самих авторов AFLNet
(комментарий в aflnet.h: добавить функцию и подключить через -P
в afl-fuzz.c).

Реализованы (patches/0002 — aflnet.c/aflnet.h, patches/0003 — aflnet-replay.c):

- `extract_requests_irc` — идентична extract_requests_ftp: IRC-команды
  тоже разделены CRLF (0x0D 0x0A), логика деления на регионы одинакова.
- `extract_response_codes_irc` — написана с нуля. У FTP код ответа —
  первые 3 байта строки ("220 ..."). У IRC код — второе слово строки
  вида ":servername 001 nick ...". Для строк, начинающихся с ':',
  пропускаем префикс до первого пробела, затем читаем второй токен:
  если это 3 цифры — используем как numeric-код напрямую; если это
  текстовая команда (JOIN, PRIVMSG, NOTICE, ERROR, PING...) — считаем
  djb2-хеш имени команды и приводим к диапазону 1000-9999 (чтобы не
  пересекаться с настоящими numeric-кодами 0-999). Так AFLNet видит
  разные команды сервера как разные состояния state machine.
- Регистрация "IRC" как значения флага -P в afl-fuzz.c (парсинг опций)
  и в aflnet-replay.c (отдельная копия той же логики для standalone-реплея).

После пересборки `-P IRC` не выдаёт ошибку unsupported protocol
(проверено: `./afl-fuzz -P IRC` без -i/-o падает на отсутствии
обязательных параметров, а не на протоколе).

## 7. Инструментирование ngIRCd и первый запуск AFLNet

**Проблема:** собственный llvm_mode AFLNet (afl-llvm-pass.so.cc)
написан под устаревший ("legacy") LLVM Pass Manager и использует
заголовок llvm/Transforms/IPO/PassManagerBuilder.h, полностью
удалённый из LLVM 15+ (у нас LLVM 21). Пересборка AFLNet-овского
llvm_mode невозможна без переписывания pass под новый Pass Manager API.

**Решение:** инструментируем target компилятором из уже установленного
AFL++ (afl-clang-fast, поддерживает LLVM 14-21), а фаззим движком
(afl-fuzz) из AFLNet — это стандартная, документированная
community-практика (инструментация AFL++ бинарно совместима с классическим
AFL forkserver protocol). Используем режим `AFL_LLVM_INSTRUMENT=CLASSIC`
для максимальной совместимости со старым протоколом форк-сервера:

```bash
cd ngircd
make distclean
AFL_LLVM_INSTRUMENT=CLASSIC CC=afl-clang-fast ./configure
AFL_LLVM_INSTRUMENT=CLASSIC CC=afl-clang-fast make -j$(nproc)
```

Собрано с инструментацией нескольких тысяч точек ветвления
суммарно по всем .c-файлам (наибольшие: irc-oper.c — 510,
log.c — 453, irc-login.c — 357).

**Проблема:** первый же dry run AFLNet падал с ошибкой
`Unable to request new process from fork server (OOM?)`.
Причина — новый рантайм инструментации AFL++ по умолчанию
делает расширенное "рукопожатие" с форк-сервером (доп. байты о
возможностях/размере карты), которое старый afl-fuzz из AFLNet
(потомок AFL 2.52b) не понимает — протокол расходится уже на
первом запросе форка.

**Решение:** переменная окружения `AFL_OLD_FORKSERVER=1`,
специально предусмотренная в AFL++ для случаев запуска
инструментированного target'а сторонним движком, ожидающим
поведения ванильного AFL:

```bash
AFL_OLD_FORKSERVER=1 ../aflnet/afl-fuzz -i seeds -o out \
  -N tcp://127.0.0.1/6667 -P IRC -D 10000 -q 3 -s 3 -E -K -R -m none \
  -- ngircd/src/ngircd/ngircd --nodaemon --config ngircd_6667.conf
```

После этого AFLNet успешно запустился, за первые секунды нашёл 3
разных пути из одного seed-файла (подтверждение, что state-aware
режим -E реально различает состояния через наш IRC-модуль).

## 8. Параллельный фаззинг (пункт 3 задания)

Каждый экземпляр AFLNet поднимает собственный процесс ngircd,
а несколько экземпляров не могут слушать один TCP-порт — поэтому
каждому инстансу (master + 3 slave) выделен отдельный конфиг
с уникальным портом (6667-6670). Запуск через tmux с 4 панелями,
см. scripts/fuzz_parallel.sh.

**Проблема:** `-S slave1 ... -d` падает с `PROGRAM ABORT: -S already
implies -d` — флаг -S (slave-режим) в AFLNet сам включает quick-and-dirty,
явно дублировать -d нельзя.
**Решение:** убрать -d из команд слейвов (только у master его не было
изначально, что и правильно — master должен пройти полный
детерминированный проход один раз на кластер).

Финальная статистика по 4 параллельным инстансам (results/out_parallel_final_stats.txt):

| instance | cycles_done | execs_done | paths_total | bitmap_cvg |
|---|---|---|---|---|
| master | 7 | 82097 | 213 | 2.24% |
| slave1 | 37 | 55454 | 265 | 2.32% |
| slave2 | 46 | 51718 | 249 | 2.29% |
| slave3 | 45 | 46730 | 246 | 2.29% |

Крашей не найдено ни на одном инстансе.

## 9. Пользовательская мутация (пункт 4 задания)

У классического AFL (и, соответственно, AFLNet 2.56b) нет
C-библиотечного API кастомных мутаторов (`afl_custom_fuzz()`) —
это фича именно AFL++. Единственный родной механизм расширения
мутаций у ванильного AFL — Python-модуль (AFL_PYTHON_MODULE),
но конкретно эта сборка AFLNet скомпилирована без поддержки Python
(в исходниках нет ни одного упоминания PYTHON).

Поэтому кастомный оператор мутации реализован прямым патчем
havoc-стадии в afl-fuzz.c (patches/0004-custom-irc-mutator.patch):
добавлен `case 21` в switch внутри havoc-цикла (после стандартных
15 базовых операций AFL + 2 dictionary-based + 4 AFLNet
region-level при -R). Новый оператор вставляет в случайную позицию
буфера случайный IRC-токен (NICK, USER, JOIN, PRIVMSG, #test, CRLF
и т.д.) вместо чисто случайных байт — это протокол-осознанная
мутация, повышающая шанс достичь кода, ожидающего конкретные
ASCII-ключевые слова (irc-login.c, irc-channel.c).

Для честного сравнения собраны два отдельных бинарника:
afl-fuzz-default (только IRC-протокол, 17 стандартных операторов)
и afl-fuzz-custom-mutator (17 + наш 18-й). Оба запущены
параллельно на разных портах, 30 минут, с одним и тем же seed.

### Результаты сравнения (30 минут)

| Метрика | Default | Custom mutator |
|---|---|---|
| cycles_done | 4 | 2 |
| execs_done | 48880 | 44010 |
| execs_per_sec | 33.77 | 30.91 |
| paths_total | 142 | 107 |
| bitmap_cvg | 2.14% | 2.07% |
| stability | 85.74% | 86.75% |
| unique_crashes | 0 | 0 |

**Честный вывод:** в этом конкретном 30-минутном прогоне дефолтные
мутации показали немного больший прирост путей и покрытия, чем
кастомный мутатор. Вероятные причины:

1. Новый оператор — лишь 1 из 18 равновероятных вариантов (~5.5%
   havoc-итераций), у AFL нет встроенного механизма повышения его веса
   (в отличие от AFL++, где есть AFL_CUSTOM_MUTATOR_ONLY).
2. Разные случайные PID-зависимые RNG seed у двух прогонов
   (cycles_done 4 vs 2) — на выборке ~45-49k короткого 30-минутного
   прогона часть разницы может быть статистическим шумом, а не
   системным эффектом мутатора.
3. Позитивный результат: stability практически одинакова
   (85.74% vs 86.75%) — вставка валидных IRC-токенов не вносит
   дополнительной нестабильности по сравнению со случайными байтами.

Корректная проверка статистической значимости потребовала бы
многократных повторов с усреднением либо кратно более длинных
сессий (часы, а не минуты) — за рамками времени, отведённого
на лабораторную работу.

## 10. LCOV HTML-отчёт о покрытии (пункт 5 задания)

### Подход

AFL-инструментация заточена под быструю обратную связь по покрытию
для самого фаззера и не даёт удобного gcov/lcov-формата напрямую.
Поэтому:

1. Собрана отдельная, третья версия ngIRCd — обычным gcc с флагами
   `--coverage -O0 -g` (без какой-либо AFL-инструментации).
2. Через `aflnet-replay` (тоже пропатчен под IRC, см. patches/0003)
   прогнан весь corpus, найденный во всех экспериментах разом:
   out_default/queue, out_custom/queue и out_parallel/{master,slave1,
   slave2,slave3}/queue — суммарно 1222 тест-кейса.
3. Каждый вызов aflnet-replay дописывает статистику в общие .gcda-файлы
   coverage-сборки (gcov суммирует счётчики при повторных запусках,
   а не перезаписывает).
4. lcov --capture + genhtml собрали финальный HTML-отчёт.

Скрипт полного цикла (старт сервера -> replay всего corpus'а ->
аккуратная остановка через SIGTERM/SIGINT, чтобы gcov успел
сбросить счётчики на диск) — scripts/collect_coverage.sh.

### Встреченные проблемы

- **MaxConnectionsIP по умолчанию = 5** — при быстром последовательном
  replay сотен тест-кейсов с одного IP сервер начал отбрасывать новые
  соединения раньше, чем система закрывала предыдущие. Решение —
  `MaxConnectionsIP = 0` в конфиге (без лимита).
- **`aflnet-replay` сам падает с Aborted на большинстве вызовов**
  (1081 из 1222) — но не критично: по логам сервера видно, что
  подавляющее большинство запросов реально доставлены и обработаны
  (registered/JOIN/PRIVMSG проходят), падает именно клиент aflnet-replay
  уже после отправки данных (вероятно, известная проблема совместимости
  кода 2020 года с современным glibc/_FORTIFY_SOURCE=2). Для цели
  "покрытие сервера" это не важно — важно, что сервер обработал вход.
- **Потеря .gcda при неаккуратном kill -9** — дважды прогон обрывался
  случайным SIGKILL сервера (из истории команд в интерактивном
  терминале), после чего накопленные счётчики покрытия терялись
  безвозвратно (SIGKILL не даёт выполниться atexit-хендлеру gcov).
  Решение — единый самодостаточный скрипт (collect_coverage.sh),
  который сам стартует сервер, сам прогоняет весь corpus, сам
  корректно останавливает через SIGTERM (с фоллбэком на SIGINT),
  без ручного вмешательства в процессе.

## 11. Итоги и краши (пункт про разбор найденных крешей)

За все прогоны (одиночный, параллельный x4, сравнение мутаторов)
уникальных крешей и хэнгов не найдено (unique_crashes: 0 везде).
Это ожидаемый результат для зрелого, активно поддерживаемого
с 2001 года проекта за ограниченное суммарное время фаззинга
(порядка 2-3 часов на всех инстансах вместе). Для повышения
шанса найти реальные баги следующим шагом было бы:

- расширить seed-корпус командами MODE/OPER/CAP/KICK (см. раздел 10)
  для роста покрытия за пределы уже достигнутых 30.8%;
- увеличить вес кастомного мутатора либо перейти на AFL++ для
  использования полноценного C custom_mutator API вместо патча
  havoc-стадии;
- увеличить длительность прогона на порядки (десятки часов вместо минут).
